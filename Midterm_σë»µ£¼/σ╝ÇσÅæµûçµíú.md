# Midterm Kaggle 开发文档

## 1. 项目目标
本次期中项目的任务是：给定自然语言 prompt，生成可解析、可渲染、满足约束的 SVG 字符串，并通过 Kaggle Code Competition 的方式提交 `id,svg` 格式结果。

核心目标不是只让模型“画得像”，而是同时做到：

- 输出 SVG 必须合法
- 推理流程必须可在 Kaggle 上无联网复现
- 训练、验证、推理代码必须可复现
- 最终还要提交 ACL 格式报告、GitHub 仓库和模型权重链接

## 2. 考试与竞赛硬性要求

### 2.1 提交形式
- 只能使用 Kaggle Code Submission，不能手动上传文件。
- 输出文件必须是 CSV，列名为 `id,svg`。
- 每队每天最多 `3` 次提交。
- 比赛期间至少提交一次。

### 2.2 报告要求
报告总分 `35/100`，需要使用 ACL 模板，并提交到 Gradescope。报告必须包含：

- Introduction: 问题定义、动机、方法概述
- Dataset: 数据集说明、预处理、增强
- Model: 模型结构和选择理由
- Experimentation: 超参数、训练设置、尝试过的方法
- Results: 定量结果、对比、消融
- Conclusion: 总结、失败经验、未来工作

报告还必须包含：

- GitHub 仓库链接
- 模型权重链接
- 尝试过哪些方法、哪些有效、哪些无效
- 是否使用过 AI 工具辅助编码/调试

### 2.3 代码要求
代码总分 `15/100`，重点看：

- 训练代码是否清晰
- 验证与推理代码是否清晰
- 是否可复现

需要额外提供：

- 最终 notebook 和/或 python 脚本
- 依赖版本列表，如 `requirements.txt`
- 固定随机种子
- 清晰复现说明

### 2.4 SVG 输出约束
- 画布大小：`256 x 256`
- 最大 SVG 长度：`16000` 字符
- 最大 path 数量：`256`
- 必须是单个根 `<svg>` 元素
- 禁止使用：
  - `script`
  - 事件处理器
  - 动画
  - `foreignObject`
  - 外部引用

允许元素包括：

- `svg`
- `g`
- `path`
- `rect`
- `circle`
- `ellipse`
- `line`
- `polyline`
- `polygon`
- `defs`
- `use`
- `symbol`
- `clipPath`
- `mask`
- `linearGradient`
- `radialGradient`
- `stop`
- `text`
- `tspan`
- `title`
- `desc`
- `style`
- `pattern`
- `marker`
- `filter`

### 2.5 评分机制
每条样本先过有效性闸门，再算质量分。

第一阶段 Validity Gate：

- SVG 能成功解析
- 标签在白名单中
- 可成功渲染
- 不超过复杂度限制

任一失败，该样本得分直接为 `0`。

第二阶段 Quality Score：

- Visual Fidelity，权重 `0.85`
  - `0.7 * SSIM + 0.3 * EdgeF1`
- Structural Similarity，权重 `0.12`
  - 基于 SVG 结构树编辑距离
- Compactness，权重 `0.03`
  - 鼓励长度接近参考答案

最终意味着：

- “先保合法，再追求质量” 是第一原则
- 无效 SVG 比画得不够像更致命
- 结构相似性和紧凑性也有分，但主要分数来自视觉一致性

### 2.6 可用与不可用资源
允许：

- 提供的训练数据
- 公开的预训练模型
- 开源库
- LLM 辅助编码和调试
- Kaggle 允许的资源

不允许：

- 外部数据集
- 用测试集做训练
- 跨队共享代码
- 为隐藏测试手工写答案
- 泄露的私有 prompt 或参考答案
- 恶意或不安全 SVG

## 3. `svg-and-lora-tutorial.ipynb` 的关键信息

这个 notebook 主要做了两件事：先讲 SVG，后讲 LoRA/QLoRA 微调。

### 3.1 SVG 部分的要点
SVG 是 XML 文本，因此可以被 LLM 当成“代码生成任务”处理。教程重点讲了这些基础：

- `svg` 是根容器，负责定义 `viewBox`、宽高
- 常用图元包括 `rect`、`circle`、`ellipse`、`polygon`、`text`、`g`
- 最重要的是 `path`

`path` 的关键命令：

- `M`: 移动到
- `L`: 直线
- `H` / `V`: 水平 / 垂直线
- `C`: 三次贝塞尔曲线
- `Q`: 二次贝塞尔曲线
- `A`: 圆弧
- `Z`: 闭合路径

教程通过简单图形和一个完整的 Pikachu SVG 说明：

- 复杂图像是由基础图元逐层组合出来的
- 合理使用 `ellipse`、`polygon`、`path`、渐变和透明度，可以显著提升表现力
- SVG 源码长度和 token 数之间存在直接关系，因此紧凑表达很重要

### 3.2 渲染与可视化
教程使用：

- `IPython.display.SVG`：在 notebook 中直接显示 SVG
- `cairosvg.svg2png(...)`：把 SVG 栅格化为 PNG

这与比赛评测非常一致，因为评测本身会先渲染，再做图像层面的比较。

开发启示：

- 本地验证必须加入 SVG 渲染环节
- 不能只检查字符串格式，还要检查渲染是否成功

### 3.3 LoRA / QLoRA 的理论重点
教程强调：

- 全量微调成本太高
- LoRA 将权重更新写成低秩分解 `Delta W = BA`
- 原始权重冻结，只训练低秩适配器
- QLoRA 进一步把底座模型量化到 4-bit，从而显著降低显存占用

关键超参数：

- `r`
- `lora_alpha`
- `lora_dropout`
- `target_modules`

推荐的 `target_modules` 包括：

- `q_proj`
- `k_proj`
- `v_proj`
- `o_proj`
- `gate_proj`
- `up_proj`
- `down_proj`

教程给出的核心判断是：对于本次比赛，QLoRA 非常适合做一个可训练、可复现、算力成本可控的 baseline。

### 3.4 微调代码骨架
教程演示了完整的 QLoRA 标准流程：

1. 加载 tokenizer 和底座模型
2. 使用 `BitsAndBytesConfig` 以 4-bit 方式加载
3. `prepare_model_for_kbit_training`
4. 配置 `LoraConfig`
5. 用 `get_peft_model` 挂载 LoRA
6. 把 `(prompt, svg)` 格式化成 instruction tuning 样本
7. 用 `TrainingArguments` + `SFTTrainer` 训练
8. 保存 adapter
9. 训练后 `merge_and_unload()` 合并权重用于推理

开发启示：

- 训练数据必须整理成严格一致的对话模板
- 推理时尽量和训练模板保持一致
- LoRA adapter 可以单独保存，适合实验管理
- 最终推理阶段可合并权重以减少额外开销

## 4. `starter-notebook.ipynb` 的代码结构与教授提示

这个 starter notebook 不是最终答案，而是一个“可以跑通的脚手架”。它的重点是给出 Kaggle 可落地的最小闭环。

### 4.1 模型与依赖
starter 采用的是 Unsloth 路线，核心配置是：

- 模型：`unsloth/Qwen3.5-2B-Instruct-bnb-4bit`
- 目标：Qwen 2B 级别 LoRA 微调
- 训练长度：`max_seq_length = 2048`
- 优化器：`paged_adamw_8bit`

这说明教授更偏向：

- 中小尺寸代码 / 指令模型
- 4bit 加载
- LoRA 微调
- 在 Kaggle 资源限制下完成训练与推理

### 4.2 数据集组织方式
starter 把多个公开 SVG 数据源放进一个 `DATASET_CATALOG` 中，并为每个数据源指定：

- `split`
- 可能的 prompt 字段名
- 可能的 svg 字段名

然后通过统一函数：

- 自动挑选第一个非空 prompt 字段
- 自动挑选第一个非空 svg 字段
- 过滤掉不以 `<svg` 开头的样本

开发启示：

- 教授明确预期你会整合多个来源的数据
- 但不同数据集字段名不统一，必须自己做 schema 归一化
- 首个版本应该先只用 `1-2` 个数据源，降低工程复杂度

### 4.3 训练样本格式
starter 用的是一个明确的 统一 prompt 模板（训练 + 推理完全一致）：

<|system|>
You are an SVG generator. Output only valid, compact SVG.

<|user|>
{prompt}

<|assistant|>
{svg}

这说明训练目标是标准的 SFT，而不是额外的 RL 或复杂多阶段训练。

文档层面的重要结论：

- baseline阶段禁止频繁修改 prompt 模板！
- system prompt 要稳定
- assistant 端只能放 SVG，不要夹杂解释文字
- 训练和推理最好使用同一模板

### 4.4 训练配置
starter 的默认参数大致是：

- `lora_r = 16`
- `lora_alpha = 16`
- `learning_rate = 2e-4`
- `per_device_train_batch_size = 2`
- `gradient_accumulation_steps = 8`
- `num_train_epochs = 1`
- `eval_size = 0.02`

这是一组明显偏 baseline 的安全配置，目的不是极致性能，而是先保证：

- 能训练
- 能收敛
- 能在 Kaggle 预算内完成

### 4.5 推理与提交生成
starter 中已经给出了一条很重要的推理链路：

- 用正则从模型输出中提取 `<svg>...</svg>`
- 用 `xml.etree.ElementTree` 做解析校验
- 无法解析时，回退到一个固定的 `fallback_svg`
- 遍历测试集 prompt，生成 `submission.csv`

这部分非常关键，因为它直接对应比赛规则中的 Validity Gate。

换句话说，starter 的核心思想不是“只追求最强生成”，而是：

- 优先确保每一条都能交出合法 SVG
- 对坏输出做兜底
- 避免整条样本拿零分

### 4.6 教授在 starter 中给出的隐含提示
从 notebook 内容看，教授实际在提示你以下策略：

- 先做一个稳定 baseline，再谈提分
- 固定随机种子
- 记录训练日志
- 统计 invalid generation / fallback 次数
- 控制训练时长，尽量在 `6-8` 小时量级内
- 根据最新上游数据集字段和模型 ID 做必要修正
- 对齐最新的 Unsloth / Qwen prompt formatting 模板

另外，starter 的一次数据加载输出里出现了大数据集下载与 kernel crash 风险，这说明：

- 不能盲目把超大数据源整包拉下来
- 最好优先使用子集、采样或 Kaggle 打包数据
- 数据读取必须尽量保守，先验证可跑通

## 5. 推荐开发路线

### 5.0 Debug 优先级（按顺序排查）：

1. SVG 是否能解析（xml）
2. 是否包含非法标签
3. 是否缺失 <svg> 根节点
4. 是否被截断（max_new_tokens）
5. prompt 模板是否一致
6. fallback 是否触发过多
7. decode 参数是否过激（temperature/top_p）

### 5.1 第一阶段：建立可提交 baseline
目标：先保证能在 Kaggle 上产出合法提交。

建议动作：

- 直接基于 `starter-notebook.ipynb` 改
- 保留 `extract_svg`、`is_valid_svg`、`fallback_svg`
- 跑通读取测试 prompt 并生成 `submission.csv`
- 即使模型效果一般，也先完成第一次提交

Baseline 成功标准（必须全部满足）：

1. Kaggle notebook 能完整运行（无报错）
2. submission.csv 正确生成
3. SVG valid rate ≥ 95%
4. fallback rate ≤ 10%
5. 平均 SVG 长度 < 5000（避免爆长度）

Baseline 阶段禁止事项：

- 不更换模型（先用 Qwen baseline）
- 不引入复杂 loss（render loss 等）
- 不使用全部数据集（先用小子集）
- 不优化结构分（先保 valid）

Baseline 最小配置：

- 数据：≤ 50k samples
- epoch：1
- r=16
- batch_size=2
- gradient_accumulation=8
- max_new_tokens=512

### 5.2 第二阶段：整理训练数据
目标：构建高质量 `(prompt, svg)` 数据。

建议动作：

- 先只保留 `1-2` 个质量较高的数据源
- 统一字段名为 `prompt` 和 `svg`
- 过滤明显无效或过长 SVG
- 增加约束过滤：
  - 长度超过 `16000` 的样本丢弃
  - path 数量过多的样本丢弃
  - 包含禁用标签的样本丢弃

推荐额外做的清洗：

- 统一 `width/height/viewBox`
- 删除无关 metadata
- 规范颜色写法
- 检查 XML 可解析性

### 5.3 第三阶段：QLoRA 微调
目标：得到一个能稳定输出合法 SVG 的 adapter。

建议起始配置：

- 底座：Qwen 2B 左右的 instruct/coder 模型
- `r = 16`
- `alpha = 16` 或 `32`
- `lr = 2e-4`
- `max_seq_length = 2048`
- 先训练 `1` 个 epoch 做烟雾测试

重点观察：

- loss 是否稳定下降
- 是否频繁输出非 SVG 文本
- 是否出现截断
- 显存和训练时长是否在预算内

### 5.4 第四阶段：离线验证
目标：不要只看训练 loss，要看提交质量。

建议做一个本地验证脚本，至少统计：

- SVG 解析成功率
- 渲染成功率
- 平均长度
- 平均 path 数量
- fallback 比例

如果时间允许，再增加：

- SSIM
- 边缘相似度
- 结构树相似度近似指标

### 5.5 第五阶段：推理优化
目标：减少无效输出，提高 leaderboard 稳定性。

可尝试方向：

- 调整 `temperature`、`top_p`
- 减少冗长输出
- 强化 system prompt 中“只输出 SVG”
- 在生成后加后处理：
  - 截取首个合法 `<svg>...</svg>`
  - 修正常见闭合错误
  - 过滤禁用标签

重要原则：

- 后处理必须保守，不能引入新的非法结构
- 宁可回退到简单合法 SVG，也不要提交不可渲染结果

### 5.6 第六阶段：整理复现与报告材料
最终必须同步整理：

- GitHub 代码仓库
- 模型权重存储链接
- `requirements.txt`
- 训练与推理说明
- 报告中的实验表格和结论

## 6. 建议的实验优先级

优先做：

1. 跑通 starter baseline
2. 保证合法 SVG 输出
3. 小规模数据微调
4. 提交并观察 public leaderboard

第二优先级：

1. 数据清洗和筛选
2. 调整 LoRA 超参数
3. 调整 prompt 模板
4. 优化后处理和 fallback 策略

最后再做：

1. 多数据源混合
2. 更复杂模型
3. 更重的训练
4. 消融实验

## 7. 当前最可执行的实现方案

如果要以最低风险完成这次 midterm，建议采用下面的方案：

1. 以 `starter-notebook.ipynb` 为主线，不重写整体框架。
2. 先选一个较小、字段明确、质量较好的 SVG 数据源。
3. 用 QLoRA 对 Qwen 2B 级模型做一轮短训练。
4. 保留强校验和 fallback 机制，优先保证合法率。
5. 在 Kaggle 上先完成可复现提交，再迭代提分。
